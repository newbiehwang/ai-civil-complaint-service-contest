1. Executive Summary

층간소음 민원은 "어디에 어떻게 접수해야 하는가"라는 경로 탐색 문제와 "증거가 부족해서 보완요청 또는 반려를 반복한다"는 자료 준비 문제, 그리고 접수 이후 "현재 어디까지 처리됐는지 알기 어렵다"는 추적 단절 문제가 한꺼번에 얽혀 있는 대표적인 비정형 민원이다. 따라서 이 민원은 단순 FAQ 챗봇보다 훨씬 높은 수준의 시나리오 설계와 상태관리, 증거 처리, 대외기관 연계 전략이 필요하다.

제안 서비스는 국민의 첫 발화 한 문장만으로 민원을 시작한다. 이후 AI는 최소 질문으로 상황을 정리하고, 위험 여부를 선분기하고, 관리사무소, 이웃사이센터, 국민신문고, 분쟁조정 등 후보 경로를 추천하며, 접수 전 필요한 소음일지와 녹음 자료를 구조화한다. 제출 직전에는 사용자 동의와 본인확인을 거치고, 제출 이후에는 상태 변화, 보완 요청, 지연 상황을 추적해 다음 행동을 안내한다.

핵심 가치 제안

가치 현재 문제 서비스가 해결하는 방식
경로 탐색 비용 제거 국민이 관리사무소, 이웃사이센터, 국민신문고, 경찰 등 여러 채널을 직접 탐색해야 함 상황 기반 경로 추천과 대안 비교, 이유 공개형 카드 제공
증거 기반 접수 성공률 향상 소음일지, 녹음, 발생시점 정보 누락으로 보완 요청 반복 대화에서 일지를 자동 생성하고 체크리스트로 충분도 판단
단절 없는 추적 접수 이후 처리 상태를 알기 어렵고 같은 설명을 반복 상태머신과 타임라인으로 단계 전환, 보완 요청, 종결까지 안내
신뢰 가능한 AI 민원 분야는 잘못된 안내의 비용이 큼 Rule-First 설계, Human-in-the-Loop, 감사로그, 폴백 경로 유지

현재 코드베이스 기준 의미

이 제안은 순수 아이디어 단계에 머물지 않는다. 현재 저장소에는 이미 Spring Boot 기반 상태머신, JWT 인증, Idempotency 처리, 증거 체크리스트, 기관 제출 mock 워커, 보완요청 라운드트립, HWPX 기반 측정신청서 생성, Expo/Flutter 클라이언트, 인터페이스 계약 문서가 존재한다. 즉, 기획의 핵심 논리를 PoC 수준으로 증명할 수 있는 기반이 마련되어 있다.

2. 문제 정의와 정책적 배경

2.1 왜 층간소음이 적합한 대표 시나리오인가

층간소음 민원은 하나의 이슈처럼 보이지만 실제로는 생활소음, 위험 상황, 공동주택 관리 이슈, 공공기관 상담, 민원 접수, 분쟁조정, 사후 추적이 결합된 복합 민원이다. 이 복합성을 해결하는 구조를 만들면 다른 생활불편 민원으로의 수평 확장이 매우 쉽다. 반대로 이 시나리오를 해결하지 못하면 범정부 확장성을 주장하기 어렵다.

2.2 사용자 관점 As-Is 불편

불편 항목 현재 국민 경험 행정적 손실
경로 탐색 비용 검색과 전화 문의를 반복하며 어느 기관이 맞는지 스스로 판단해야 함 잘못된 채널 접수, 기관 이관, 재문의 증가
증거 준비 불확실성 소음일지 작성법, 녹음 기준, 첨부 방법을 사전에 알기 어려움 보완 요청, 반려, 처리 지연
조정 실패 후 다음 단계 공백 관리사무소 또는 1차 조정이 실패해도 다음 행동을 다시 탐색해야 함 민원 중도 이탈, 감정적 갈등 심화
진행상태 단절 처리중인지, 보완 요청이 왔는지, 누가 담당인지 파악하기 어려움 문의 반복, 응답 지연 체감 확대
위험 상황 혼재 폭행, 협박, 스토킹이 동반돼도 일반 생활소음처럼 접근할 수 있음 안전 리스크 증가, 오분류 비용 확대

2.3 기존 채널과의 차이

기존 채널 잘하는 일 부족한 일 AI 서비스 차별점
관리사무소 단지 내 초기 전달과 조정 증거 작성 지원, 실패 후 다음 경로 안내, 디지털 추적 부족 조정 실패 후 다음 경로로 연속 전환
이웃사이센터 상담, 현장 조정, 소음측정 안내 사용자 입장에서는 사전 준비와 요건 판단이 여전히 어려움 요건 충족도를 미리 평가해 접수 가능성 제고
국민신문고 공식 민원 접수와 처리 기록 적정 관할과 서류 완성도를 사전에 보장하지 않음 분류-라우팅-증거패키지를 선행해 제출 성공률 개선
112 즉시 위험 대응 생활소음 자체 해결 절차는 아님 위험 징후는 즉시 분기하고 그 외는 별도 민원 흐름 유지

2.4 정책 및 서비스 혁신 포인트

  • 검색 중심 전자정부 경험을 해결 중심 대화형 경험으로 전환한다.
  • 민원 제출 이전 단계의 준비 행위를 서비스 범위에 포함해 실제 성공률을 높인다.
  • 한 번의 접수 이후 조정, 신고, 보완, 완료까지 케이스 단위로 추적 가능한 구조를 만든다.
  • 민원 응대 데이터를 축적해 향후 반복 민원 핫스팟과 제도 개선 근거로 확장한다.

3. 서비스 시나리오와 사용자 여정

3.1 서비스 한 줄 정의

이 서비스는 "윗집 소음 때문에 힘들어요"라는 자연어 한 문장만으로 시작해서, AI가 상황을 분류하고, 경로를 추천하고, 증거를 정리하고, 제출 후 진행을 추적하는 층간소음 민원 원스톱 에이전트다.

3.2 핵심 페르소나

페르소나 상황 주요 니즈 AI가 제공하는 가치
신혼부부 야간 소음으로 수면 장애, 관리사무소 조정 실패 다음 단계 안내, 증거 준비, 신속한 진행 조정 후 공식 절차까지 단계별 로드맵 제시
고령 독거 어르신 디지털 기기 사용 어려움, 전화 또는 음성 의존 쉬운 접수, 서류 대필, 최소 입력 음성 중심 접수와 자동 문서화
다세대 주택 집주인 세입자 갈등 중재와 책임 범위 고민 사실 중심 안내, 조정 초안, 법적 범위 설명 분쟁 감정 완화형 초안과 절차 안내

3.3 8단계 사용자 여정

  1. 접수: 사용자가 자연어로 상황을 설명하면 새 케이스가 열린다.
  2. 분류: 위험 여부, 거주 형태, 시간대, 빈도, 관리주체 여부를 최소 질문으로 수집한다.
  3. 경로 설계: 관리사무소, 이웃사이센터, 국민신문고, 분쟁조정 중 적합 경로를 추천한다.
  4. 증거 수집: 소음일지, 녹음, 메타데이터, 발생시각을 구조화한다.
  5. 조정 지원: 필요한 경우 관리사무소 전달문이나 조정 요청문을 AI가 초안으로 작성한다.
  6. 정식 제출: 제출 직전 사용자 동의와 본인확인을 거쳐 기관 제출 또는 수동 제출용 문서를 생성한다.
  7. 진행 추적: 접수번호와 타임라인을 기준으로 처리 상태와 보완 요청을 추적한다.
  8. 종결 및 재발 대응: 처리 결과를 요약하고 재발 시 재활용 가능한 증거 패키지를 유지한다.

3.4 사용자 여정 상세 시나리오

가장 대표적인 흐름은 다음과 같다. 사용자가 "밤마다 윗집이 쿵쿵거린다"고 입력하면 서비스는 먼저 지금도 소음이 나는지, 위험 상황이 동반됐는지, 아파트/빌라/오피스텔인지 등을 확인한다. 이후 관리사무소 조정 여부와 방문상담 이력 여부를 확인한 뒤, 현재 단계에서 가장 성공 가능성이 높은 경로를 추천한다. 이때 서비스는 하나의 정답만 강요하지 않고, 추천 경로와 대안 경로를 함께 제시하며 각각의 장단점과 리스크를 설명한다.

사용자가 추천을 확정하면 증거 수집 단계로 들어간다. 증거는 단순 파일 업로드가 아니라 "기관이 실제로 요구하는 형태로 쓸 수 있는 패키지"여야 한다. 따라서 AI는 소음일지 항목을 자동 구성하고, 녹음 메타를 정리하며, 제출 전 부족한 항목을 체크리스트로 알려준다. 증거가 부족할 경우 즉시 제출도 허용하되 보완 요청 가능성을 명시적으로 고지한다.

3.5 예외 경로 설계

예외 상황 서비스 반응 왜 중요한가
폭행, 협박, 보복 우려 생활소음 흐름을 중단하고 112 또는 보호 안내를 우선 제시 안전 이슈는 편의보다 보수성이 우선
관리주체 없음 관리사무소 경로를 제거하고 다른 공적 채널을 앞세움 잘못된 1차 조정 권고 방지
기관 연계 장애 자동 제출 대신 HWPX/PDF 초안 생성과 수동 제출 가이드 제공 서비스 실패를 접수 실패로 연결하지 않기 위함
증거 부족 추가 기록 권장 또는 지금 제출 시 리스크 고지 후 선택권 부여 신뢰와 자율성 유지

4. AI 서비스 단계별 역할 정의

4.1 설계 원칙: Rule-First, AI-Assist, HITL

민원 서비스에서 AI는 전부를 자동화하는 존재가 아니라, 사용자의 행정 수행을 보조하되 실수를 줄이는 방향으로 배치되어야 한다. 따라서 본 서비스는 세 가지 원칙을 따른다. 첫째, 안전과 절차 분기는 Rule-First로 설계한다. 둘째, 자연어 이해와 초안 생성처럼 인간 언어 처리 가치가 큰 영역에 AI를 집중한다. 셋째, 제출과 같은 민감 행위는 Human-in-the-Loop를 거친다.

4.2 단계별 AI 역할 매트릭스

서비스 단계 AI 역할 처리 방식 출력물
접수 사용자 의도 인식, 민원 시작 여부 판단 LLM + 룰 초기 응답, 접수 유도, 새 케이스 생성
분류 slot filling, 위험 여부, 거주 형태, 빈도 정리 룰 기반 follow-up + 선택적 LLM 필수 슬롯, 리스크 신호, 다음 질문
경로 설계 복합민원 분해와 경로 추천 Rule 상태머신 + 라우팅 규칙 추천 채널 목록, 이유, 필요 증거
증거 수집 소음일지 구조화, 충분도 평가 LLM 구조화 + 룰 체크리스트 Evidence Pack, 부족 항목, 제출 가능 상태
조정 지원 사실 중심 조정 요청문 초안 LLM + 템플릿 관리사무소 전달문, 조정 요청 초안
정식 제출 서식 채움, 첨부 구성, 사용자 확인 보조 자동화 + HITL 제출 payload 또는 수동 제출 문서
진행 추적 상태 변화 요약, 보완 요청 해석 상태머신 + 필요 시 LLM 요약 사용자용 상태 메시지, 다음 행동 안내
종결 결과 요약, 재발 대응 추천 LLM 요약 + 룰 종결 요약 카드, 재신청 준비 안내

4.3 기능별 AI 적용 적합성

기능 Rule 중심 AI 중심 판단 이유
위험 키워드 감지 높음 낮음 오분류 비용이 높기 때문에 보수적 규칙 우선
자연어 슬롯 추출 중간 높음 비정형 서술에서 시간, 빈도, 감정, 맥락 추출 필요
관할 기관 추천 높음 중간 법령과 운영 규칙 매트릭스를 우선 적용하고 애매한 케이스만 AI 보조
증거 충분도 안내 높음 중간 기관별 접수 요건은 룰로 명시하고 설명 문장만 AI 보조 가능
초안 작성 낮음 높음 사용자 서술을 공문형 문장으로 바꾸는 생성 가치가 큼
상태 요약 알림 중간 중간 상태값 자체는 룰, 사용자 문장화는 AI 활용 가능

4.4 멀티 에이전트 협업 구조

목표 아키텍처에서는 하나의 거대 모델이 모든 판단을 하지 않는다. 오케스트레이터가 전체 케이스 상태를 관리하고, 그 아래에 의도 파악, 라우팅, 증거 패키지화, 초안 생성, 알림 요약을 수행하는 전문 에이전트를 둔다. 각 에이전트의 출력은 proposal, confidence, evidence, constraints 형식으로 표준화하고, 최종 판단은 오케스트레이터가 하드 룰과 상태 전이를 기준으로 결정한다.

사용자 입력 -> Case Orchestrator -> Rule Engine(위험/관할 1차 분기) -> Intake/Intent Agent(필수 슬롯 수집) -> Routing Agent(추천 경로 계산) -> Evidence Agent(증거 패키지/충분도) -> Draft Agent(문서 초안 생성) -> Notify Agent(상태 메시지 요약) -> 사용자 확인/HITL -> 기관 제출 또는 수동 제출 폴백

4.5 사용자 경험 측면의 설명 가능성

AI가 경로를 추천할 때는 단순히 "이웃사이센터로 가세요"라고 끝나면 안 된다. 추천 채널, 추천 이유, 현재 갖춘 증거, 부족한 자료, 다른 선택지, 지금 제출했을 때의 리스크를 함께 보여줘야 한다. 이 설명 가능성은 단지 윤리 요구사항을 만족시키기 위한 것이 아니라, 실제 사용자의 수용성과 신뢰도를 높여 중도 이탈을 낮추는 핵심 UX 원칙이다.

5. 기술 아키텍처 전체상

5.1 목표 아키텍처 개요

목표 아키텍처는 국민 접점 채널, 인증과 동의 처리, 케이스 오케스트레이터, AI 계층, 지식 계층, 기관 연계 계층, 데이터 저장 계층, 감사와 모니터링 계층으로 분리된다. 이렇게 역할을 나누는 이유는 민원 처리의 책임 경계와 장애 대응 경로를 명확하게 하기 위해서다.

모바일/웹/상담 UI -> API Gateway / 인증 / 동의 -> Case Orchestrator / 상태머신 / SLA 관리 -> Rule Engine + AI Agents -> RAG / 지식DB / 요건 매트릭스 -> MCP Hub 또는 직접 API 연계 -> 통합 민원 DB / 증거 저장소 / 감사로그 / 이벤트 버스

5.2 계층별 역할

계층 주요 역할 현재 구현 상태 확장 방향
프론트엔드 대화, 선택형 인터페이스, 증거 등록, 상태 확인 Expo 앱, Flutter 앱, 웹 스캐폴드 존재 Gov24 UX와 통합, 실사용성 개선
API / 오케스트레이션 케이스 생성, 상태 전이, 라우팅, 제출, 타임라인 관리 Spring Boot 서비스로 구현 실기관 연계, 운영 SLA, 분산 작업 큐 강화
AI 계층 대화 계획, 슬롯 추출, 초안 생성, 요약 Rule-based fallback과 Claude/OpenAI 클라이언트 포함 RAG, 오디오 ML, 고도화된 정책 모델 추가
데이터 계층 케이스, 증거, 타임라인, 사용자, 중복 제출 방지 기록 저장 Postgres + JPA + Flyway 스키마 포함 오브젝트 스토리지와 분석 파이프라인 추가
기관 연계 계층 기관 제출, 상태 조회, 장애 폴백 Mock worker와 제출 채널 enum 수준 구현 MCP Hub 또는 기관별 어댑터 실연동
보안 계층 JWT 인증, 동의, PII 통제, 감사로그 JWT 기반 보호와 기본 오류 응답 구현 실사용 인증, 데이터 마스킹, 접근 추적 강화

5.3 현재 저장소에서 확인되는 핵심 구조

현재 백엔드는 Java 21 기반 Spring Boot 3.3.6으로 작성되어 있으며, 웹, 검증, 보안, OAuth2 resource server, JPA, Flyway, Mail 의존성을 포함한다. 구조는 API 계층, application 계층, persistence 계층, config 계층으로 분리되어 있다. 이는 문서상 제안하는 계층형 아키텍처와 일치하며, 향후 기관 연계나 RAG 기능을 추가하기에 적절한 뼈대다.

5.4 상태머신 중심 설계

민원 서비스에서 가장 중요한 것은 예외 없는 상태 관리다. 현재 코드베이스는 RECEIVED, CLASSIFIED, ROUTE_CONFIRMED, EVIDENCE_COLLECTING, FORMAL_SUBMISSION_READY, INSTITUTION_PROCESSING, SUPPLEMENT_REQUIRED, COMPLETED, CLOSED 등의 상태 전이를 명시적으로 관리하고, 허용되지 않은 전이는 409 오류로 차단한다. 이 구조 덕분에 사용자가 어떤 단계에 있는지, 제출 가능 여부가 무엇인지, 보완 요청을 왜 받아야 하는지가 일관되게 설명된다.

6. 세부 기술 설계: 모델, 데이터, 연계, 보안

6.1 모델 아키텍처

영역 현재/권장 모델 전략 역할 운영 원칙

follow-up 질문 Rule-based 우선, 선택적 OpenAI client 필수 슬롯을 채우기 위한 최소 질문 생성 접수 초기 단계는 룰 기반으로 고정

6.2 데이터 아키텍처

현재 데이터 모델은 민원 케이스, 증거 항목, 타임라인 이벤트, 데모 사용자, 중복 제출 방지 레코드로 구성된다. 이 구성이 좋은 이유는 "민원 본문"보다 "민원 처리 과정"을 중심으로 데이터를 저장한다는 점이다. 즉, 케이스가 있고, 그 케이스에 증거와 이벤트가 축적되며, 제출의 재시도와 사용자 식별이 부가된다.

데이터 객체 현재 저장 필드 의미 향후 확장
Complaint Case 시나리오, 주거형태, 상태, 리스크, filled slots, 라우팅, current action 민원 케이스의 단일 진실 원천 SLA, 관할기관, 제출 결과, 익명화 정책
Evidence Item 유형, 저장키, 파일명, capturedAt, adequacy score 증거 등록과 충분도 평가의 기반 해시, 오디오 분석 결과, 위치 메타 추가
Timeline Event 이벤트 유형, 시각, 제목, 설명, actor 사용자 추적과 감사의 핵심 로그 SLA breach, 담당자, 재시도 이력
Idempotency Record operation, key, request/response JSON 생성/제출 중복 방지 TTL, 보안 마스킹, 감사 연동
App User username, password hash, profile 데모 로그인과 사용자 프로필 채움 Gov24 SSO, MFA, 권한 모델

6.3 데이터 파이프라인과 지식계층

본 서비스는 운영 데이터와 지식 데이터를 분리해서 관리해야 한다. 운영 데이터는 개별 민원 케이스와 사용자의 개인정보, 증거 메타데이터, 기관 처리 이력이다. 반면 지식 데이터는 법령, FAQ, 기관 접수 요건, 민원 유형별 라우팅 매트릭스, 상담 스크립트와 같은 기준 정보다. 운영 데이터는 정합성과 보안이 핵심이고, 지식 데이터는 갱신성과 설명 가능성이 핵심이다.

향후 RAG 도입 시에는 법령과 FAQ를 조문/항목 단위로 청킹하고, 벡터 검색 후 exact match 검증을 거쳐 화면에 근거를 노출해야 한다. 특히 법령 인용은 LLM 생성 결과를 그대로 노출하지 않고, 검색된 원문과 일치하는지 사후 검증하는 절차를 둬야 한다.

6.4 기관 연계 아키텍처

목표 상태에서는 MCP Hub 또는 기관별 어댑터 계층을 둬서 이웃사이센터, 국민신문고, 향후 정부24, 지자체 민원 시스템과 느슨하게 연결한다. 그러나 실제 행정 시스템 연계는 기술보다 협의와 보안 심사가 더 오래 걸리는 경우가 많기 때문에, 이 문서는 연계가 지연돼도 사용자 가치가 유지되도록 자동 작성 서식과 수동 제출 폴백을 아키텍처에 포함한다.

연계 방식 장점 리스크 권장 사용 시점
MCP 기반 표준 연계 기관 추가 시 확장성이 높고 오케스트레이터 변경 범위가 작음 표준화와 기관 협력 필요 MVP 이후
기관별 직접 API 빠른 실연동 가능 기관별 상이한 보안/인증/스펙 관리 부담 특정 기관 우선 PoC
문서 자동 생성 후 수동 제출 연계가 없어도 서비스 가치 제공 가능 완전 자동화는 아님 즉시 사용 가능한 기본 폴백

6.5 보안 아키텍처

보안 항목 원칙 현재 근거 추가 과제

6.6 신뢰도 높은 AI 설계

  • 위험 분기, 상태 전이, 제출 요건은 규칙과 상태머신이 책임진다.
  • 생성형 AI는 설명 문장과 자연어 요약, 초안 작성처럼 잘하는 영역에 집중한다.
  • 법령과 요건은 RAG 검색 후 exact match 검증을 거쳐 근거를 노출한다.
  • 제출 행위는 반드시 사용자 동의와 본인확인을 요구한다.
  • 기관 연계 실패 시 자동 재시도 후 수동 제출 폴백을 제공한다.

클라이언트 Expo 앱, Flutter 앱, 웹 스캐폴드와 mock 데이터 포함 시연과 사용자 검증이 가능한 상태

7.2 테스트로 검증된 범위

현재 테스트는 단순 컨트롤러 smoke 수준이 아니다. 인증 없는 요청 차단, JWT 인증 성공, 라우팅 플로우, 증거 등록 전 상태 제약, 비동기 제출 완료, idempotency 재시도, 기관 게이트웨이 장애 시 복구 가능 상태 유지, 보완요청 round-trip까지 커버한다. 이는 심사 또는 투자 관점에서 "상태 관리와 예외 처리"를 이미 코드로 다뤘다는 강한 신뢰 근거다.

테스트 시나리오 검증 내용
Security Integration 토큰 없는 요청은 401, JWT 있으면 생성/분류/라우팅 성공
Scenario API 케이스 생성, idempotency, follow-up 질문, 비동기 완료, timeline 이벤트 확인
Supplement Flow 기관 보완 요청 -> 사용자 응답 -> 재처리 -> 완료 라운드트립
Gateway Error DIRECT_API 장애 시 FORMAL_SUBMISSION_READY 상태 유지와 재시도 가능성 보장

7.4 개발 범위 정의

본 프로젝트의 개발 범위는 크게 세 층으로 정리할 수 있다. 첫째, 사용자 접점 계층에서는 자연어 대화, 선택형 UI, 증거 첨부, 상태 피드 화면을 제공한다. 둘째, 업무 계층에서는 상태머신, 라우팅, 증거 체크리스트, 제출, 보완 대응을 관리한다. 셋째, 연계 및 운영 계층에서는 기관 접수, 알림, 감사, 지표 수집, 장애 폴백을 관리한다. 이 중 현재 저장소는 첫째와 둘째의 핵심 골격을 이미 갖추고 있고, 셋째 계층은 mock/설계 수준으로 진입해 있다.

9. 리스크 요인 및 대응 방안

9.1 리스크 관리 기본 관점

이 서비스의 리스크는 기술 실패보다 "잘못된 자동화"에서 더 크게 발생한다. 따라서 리스크 관리는 기술적 장애, 행정 연계 지연, AI 신뢰성, 개인정보 보호, 사용자 수용성, 운영장애의 여섯 축으로 분리해서 봐야 한다.

9.2 주요 리스크와 대응 전략

9.3 운영 장애 대응 원칙

  • 자동 재시도는 즉시, 단기 지연, 장기 지연의 3단계로 운영한다.
  • 장애가 지속되면 사용자는 현재 상태, 실패 이유, 대체 경로를 즉시 확인할 수 있어야 한다.
  • 사용자의 이미 입력한 정보와 생성한 문서는 보존되어야 하며, 장애로 다시 처음부터 작성하게 해서는 안 된다.
  • 기관 연계 장애는 서비스 전체 실패가 아니라 제출 방식 전환 문제로 취급한다.

9.4 리스크 관리에서 현재 코드의 장점

현재 저장소는 이미 몇 가지 핵심 리스크에 대한 방어 논리를 포함한다. 예를 들어, idempotency 처리는 모바일 네트워크 재시도에서 중복 제출을 막고, DIRECT_API 실패 시 케이스를 FORMAL_SUBMISSION_READY에 남겨 사용자가 재시도할 수 있게 한다. 보완 요청 round-trip 테스트는 기관이 추가 자료를 요구하는 실제 행정 시나리오에 대한 대응력을 보여준다.

10. 기대효과 및 성과 측정 지표(KPI)

10.1 국민 체감 효과

  • 어느 채널을 선택해야 하는지 모르는 상태에서 출발해, 실제로 다음 행동이 제시되는 경험으로 전환된다.
  • 민원 제출 전 단계에서 증거를 준비시키기 때문에 보완요청과 반려 경험이 줄어든다.
  • 처리 상태를 케이스 단위로 확인할 수 있어 "설명 반복"과 "막막함"이 줄어든다.
  • 고령자나 디지털 취약층도 음성 기반 또는 최소 입력으로 접근 가능한 구조를 만들 수 있다.

10.2 행정 효율화 효과

  • 관할 착오에 따른 이관 건수를 줄인다.
  • 동일 민원의 중복 채널 접수를 줄인다.
  • 보완 요청에 필요한 시간을 단축한다.
  • 반복 민원 지역이나 유형을 분석 가능한 데이터로 전환한다.

10.4 KPI 해석 원칙

중요한 점은 KPI를 단순 운영 수치로만 두지 않는 것이다. 예를 들어 보완요청 발생률이 낮아졌다면, 이는 곧 증거 준비 품질이 좋아졌음을 의미하고, 사용자 경험 측면에서는 좌절 비용이 줄어든 것이다. 또 경로 추천 수용률은 AI의 설명 가능성과 신뢰도가 함께 작동하는 지표이므로, 추천 이유 카드 개선과 연결해서 읽어야 한다.

11. 시연 전략과 운영 시나리오

11.1 심사 또는 발표 상황에서 보여줄 데모 흐름

  1. 사용자가 "밤 11시 이후 윗집 소음 때문에 잠을 못 잔다"고 입력한다.
  2. AI가 3개 이하 질문으로 위험 여부, 거주 형태, 조정 여부를 확인한다.
  3. 추천 경로와 이유, 대안 채널, 필요 증거를 카드로 보여준다.
  4. 사용자가 경로를 확정하면 소음일지, 녹음, 체크리스트가 생성된다.
  5. 제출 전 동의/본인확인 단계를 시뮬레이션하고 제출을 시작한다.
  6. 기관에서 보완 요청이 왔다는 이벤트를 주입하고 사용자가 응답한다.
  7. 타임라인에 SUBMISSION_STARTED, SUPPLEMENT_REQUESTED, SUPPLEMENT_RESPONDED, COMPLETED가 쌓이는 모습을 보여준다.

11.2 데모에서 특히 강조할 포인트

경로 추천과 이유 공개 설명 가능성과 신뢰성 포인트 routing recommendation과 selected option 구조
증거 체크리스트 기존 채널 대비 차별화 포인트 EvidenceChecklist와 sufficiency 계산
제출 이후 추적 원스톱 가치의 핵심 timeline, mock worker, supplement flow

11.3 운영 단계에서 필요한 화면

실서비스로 갈수록 사용자 화면 외에 운영 화면이 필요해진다. 예를 들어 기관 연계 장애 현황, 제출 지연 케이스, 보완 요청 만기 임박 케이스, 위험 분기 발생 내역, LLM fallback 비율, 라우팅 오판 피드백 등을 보는 운영 대시보드가 필요하다. 이는 기술적으로 복잡하기보다 서비스 품질 유지에 핵심이다.

12. 결론

시나리오 A는 단순한 층간소음 민원 챗봇이 아니다. 이 제안의 본질은 비정형 민원을 케이스 단위로 구조화하고, 경로 탐색과 증거 준비, 접수, 진행 추적을 하나의 상태 기반 서비스로 엮는 것이다. 즉, "무엇을 물어볼까"보다 "어떻게 끝까지 해결 경험을 설계할까"에 초점을 둔다.

현재 저장소는 이미 이 방향을 지지하는 강한 근거를 갖고 있다. 대화형 intake, 라우팅, 상태머신, 증거 체크리스트, mock 제출, 보완요청 처리, HWPX 서식 자동 생성, 모바일 클라이언트, 테스트 커버리지가 모두 존재한다. 따라서 이 문서는 추상적인 미래상이 아니라, 이미 작동하는 골격 위에서 실기관 연계와 운영 신뢰성을 확장하는 현실적인 개발 제안으로 읽혀야 한다.

최종 제안 문장

층간소음 민원은 "어디에 접수할까"의 문제가 아니라 "어떻게 끝까지 해결될 수 있게 도울까"의 문제다. AI 민원 서비스는 이 문제를 대화, 증거, 경로, 제출, 추적의 연결로 해결한다.
